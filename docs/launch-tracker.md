# Launch Tracker

Use this as the working checklist for the first KRAIL adoption push.

## Release Readiness

- [x] Complete the stable local-runtime `1.0.0` contract
- [x] Pass the package, lifecycle, nested-project, and MCP release gates
- [ ] Push the reviewed release branch
- [ ] Create and push the `v1.0.0` tag
- [ ] Confirm the GitHub release workflow publishes `krail` and `rail-mcp`

## Conversion Surface

- [x] Rewrite the README around a user-facing outcome
- [x] Clarify `krail` install vs `rail` import naming
- [x] Add launch and demo docs to the docs index
- [x] Improve MCP README positioning
- [x] Improve PyPI package description and keywords

## Demo Asset

- [x] Create a short demo script
- [ ] Record one terminal demo using the recommended script
- [ ] Export one GIF or short clip for GitHub and social posts
- [ ] Capture one screenshot that shows the workflow output clearly

## Launch Content

- [x] Draft X / LinkedIn launch copy
- [x] Draft Show HN copy
- [x] Draft Reddit / niche-community copy
- [x] Draft a blog post outline
- [ ] Turn the outline into a full post

## Feedback Loop

- [x] Write a growth plan
- [x] Add an adoption feedback issue template
- [ ] Add labels in GitHub for `feedback`, `onboarding`, and `messaging`
- [ ] Review first user feedback and cluster repeated confusion points
- [ ] Patch the top 3 confusion points before the broader launch

## Distribution

- [ ] Post the launch thread from the main account
- [ ] Share to 2 to 3 niche communities with a feedback ask
- [ ] Submit Show HN once the demo asset is ready
- [ ] Publish the first walkthrough post

## Success Signals

- [ ] At least 3 pieces of concrete onboarding feedback
- [ ] At least 1 external user reports a successful first run
- [ ] Repeated language from users matches the new positioning
- [ ] README and demo do most of the explanatory work
