# Launch Kit

This is the operating checklist for making KRAIL easier to adopt and easier to
share.

## Goal

Turn KRAIL from an interesting repo into a legible product with a repeatable
launch loop.

## Immediate Deliverables

- sharper README and PyPI messaging
- one short demo video or GIF
- one walkthrough post
- one Show HN post
- three niche-community feedback shares

## Launch Sequence

### 1. Conversion surface

Finish before doing broad promotion:

- README headline explains the user outcome
- first screen shows why KRAIL exists
- quickstart works on a clean machine
- install/import naming is explicit
- docs index links to demo and launch assets

### 2. Demo asset

Create one reusable proof artifact:

- short terminal recording following [Demo Script](demo-script.md)
- one static screenshot or GIF for GitHub and social posts
- one sentence explaining the pain solved

### 3. Primary launch post

Publish one post built around a real problem:

- pain: chat context does not scale to serious project work
- proof: KRAIL gives agents a repo-backed local workspace
- demo: capture -> search -> think -> workflow dry-run
- ask: request feedback from people building agent workflows

### 4. Community feedback round

Share with:

- MCP communities
- local-LLM communities
- AI coding assistant communities
- research tooling communities

Rule:

- ask for feedback on the workflow, not vanity metrics

### 5. Patch and relaunch

After feedback:

- fix the top 3 confusion points
- publish a short follow-up showing what changed
- then do the broader launch pass

## Messaging Rules

- lead with the problem, not the architecture
- say "local-first memory and workflows for agents" before "knowledge runtime"
- avoid assuming readers understand packs, integrity, or ontology language
- explain `krail` install vs `rail` import every time it matters
- show outputs, not only commands

## What To Measure

- GitHub page views
- README engagement proxies
- PyPI downloads
- stars after each launch event
- number of useful feedback comments
- repeated confusion points

## Owner Split

Codex can do:

- messaging, docs, demo scripts, post drafts, and launch checklists
- repo updates based on feedback

Human owner should do:

- record or approve the final demo asset
- publish from real accounts
- reply in communities
- learn from user reactions
