# Domain Agents

Each project has a **domain agent** — an AI agent scoped to that project's ontology, data sources, pipelines, and action catalog. The agent is the primary interface for research workflows: it can discover data, build pipelines, hydrate the ontology, run analysis, and produce reports entirely autonomously.

Domain agents come before a platform-wide agent. Each agent is an expert in its domain because it only sees its project's data and a tight, deterministic context snapshot. A platform-wide routing agent is a future concern.

---

## Agent Philosophy

**The agent is a program, not a chatbot.** It has a formal set of tools (typed, governed operations) and runs multi-turn loops until work is complete. The LLM provides reasoning; the ontology provides the contract; the tools execute real actions.

**LLM outputs are untrusted.** All tool arguments generated by the LLM are validated before execution. The agent's `allowed_actions` list (from `agents/config.yaml` in the project repo) is a hard constraint — the agent cannot call tools outside its catalog regardless of what the LLM requests.

**Context is engineered, not retrieved.** Before each conversation turn, the platform assembles a structured snapshot of the project state from the ontology. This is deterministic and ontology-derived — not fuzzy vector retrieval. The agent always knows exactly what data exists, what pipelines are available, and what the schema looks like.

**Provider-agnostic.** The agent uses LiteLLM and works with any supported model. Model is set per-project in `agents/config.yaml` and overridable per-session.

## Agent Configuration

Stored in `agents/config.yaml` in the project repo and synced to the Convex `projects` table.

```yaml
# agents/config.yaml
model: claude-sonnet-4-6              # LiteLLM model string
system_prompt_extra: |                # appended to the base system prompt
  This project focuses on New Jersey labor market analysis.
  Prioritize FRED unemployment series (NJURN, NJURN*) and Census ACS data.
  When generating reports, include county-level breakdowns.

allowed_actions:                      # governs which tools the agent may call
  - discover_sources
  - create_data_source
  - create_pipeline
  - run_pipeline
  - query_ontology
  - run_sql
  - execute_python
  - generate_report
  - publish_to_github

max_turns: 15                         # max tool-call turns per conversation (default: 10)
```

If `allowed_actions` is absent, all actions are permitted. Actions not in the list are silently unavailable — the LLM never sees their tool schemas.

## Context Snapshot

Before every conversation turn, the agent service assembles a **context snapshot** from the project's current ontology state. This snapshot is injected into the system prompt. It is deterministic, fast to assemble, and never stale (always reflects the current DuckDB and Convex state).

```python
# agent_service.py — context assembly
context = {
    "project": {
        "name": "NJ Economic Analysis",
        "slug": "nj-economics",
        "status": "hydrated",
        "last_hydrated": "2024-01-15T08:00:00Z"
    },
    "ontology": {
        "classes": [
            {"name": "State", "instance_count": 50},
            {"name": "County", "instance_count": 3142},
            {"name": "LaborIndicator", "instance_count": 48600},
            {"name": "HousingIndicator", "instance_count": 12300},
        ],
        "schema_ddl": "CREATE TABLE State (...); CREATE TABLE County (...); ..."
    },
    "data_sources": [
        {"slug": "nj_unemployment", "connector": "fred-observations", "last_fetched": "..."},
        {"slug": "census_counties", "connector": "census-tigerweb-counties", "last_fetched": "..."},
    ],
    "pipelines": [
        {"slug": "nj-hydration", "status": "success", "last_run": "2024-01-15T08:00:00Z"}
    ],
    "analysis_plugins": [
        {"slug": "unemployment_trends", "description": "..."},
    ]
}
```

The context snapshot is serialized into the system prompt header, before the user's message. The LLM sees a precise, structured picture of what it has access to.

## Tool Catalog

All tools follow the OpenAI function-calling format. LiteLLM normalizes them for each provider. Tools not in the project's `allowed_actions` list are excluded from the schema sent to the LLM.

### `discover_sources`
Search the shared connector template registry for data sources relevant to a topic.

```json
{
  "query": "unemployment labor market",
  "tags": ["economics", "fred"]
}
```
Returns: list of matching connector templates with slug, name, description.
Maps to: `GET /api/v1/connectors?q={query}&tags={tags}`

### `create_data_source`
Create a new API config in the project (optionally extending a connector template).

```json
{
  "name": "nj_unemployment",
  "extends": "fred-observations",
  "params": {"series_id": "NJURN", "observation_start": "2000-01-01"},
  "description": "NJ unemployment rate from FRED"
}
```
Returns: `{slug, valid, errors}`. On success, creates the config in Convex and commits to GitHub.
Maps to: `POST /api/v1/configs/apis`

### `create_pipeline`
Create a new pipeline config.

```json
{
  "name": "nj-labor-pipeline",
  "steps": [
    {"name": "load_states", "api": "census_states", "class": "State", "uri": "State_{fips}"},
    {"name": "load_unemployment", "api": "nj_unemployment", "class": "LaborIndicator", "uri": "Labor_{series}_{date}"}
  ]
}
```
Returns: `{slug, valid, errors}`.
Maps to: `POST /api/v1/configs/pipelines`

### `run_pipeline`
Trigger a hydration pipeline and stream progress until completion.

```json
{"pipeline_slug": "nj-labor-pipeline"}
```
Returns: `{job_id, status, steps_completed, duration_seconds, error?}`. Polls until `success` or `failed` (max 10 minutes).
Maps to: `POST /api/v1/jobs` then `GET /api/v1/jobs/{id}` with polling.

### `query_ontology`
List instances of an OWL class, with optional keyword search.

```json
{
  "class_name": "LaborIndicator",
  "search": "Bergen",
  "limit": 50
}
```
Returns: list of entity summaries (id, iri, properties dict).
Maps to: `GET /api/v1/ontology/classes/{class}/instances`

### `search_entities`
Full-text keyword search across all entity classes.

```json
{"query": "Bergen County unemployment", "types": ["County", "LaborIndicator"]}
```
Returns: ranked list of entity summaries.
Maps to: `GET /api/v1/ontology/search`

### `run_sql`
Execute SQL against the project's DuckDB.

```json
{
  "query": "SELECT c.hasName, AVG(l.hasValue) as avg_unemployment FROM County c JOIN measuredFor_edges mf ON mf.object_iri = c._iri JOIN LaborIndicator l ON l._iri = mf.subject_iri GROUP BY c.hasName ORDER BY avg_unemployment DESC LIMIT 10"
}
```
Returns: `{columns, rows, rowCount}`.
Maps to: `POST /api/v1/sql`

### `get_sql_schema`
Returns the DuckDB schema (all tables and columns) so the agent can write accurate queries.

```json
{}
```
Returns: `{table_name: [{name, type}]}` plus edge tables from object properties.
Maps to: `GET /api/v1/sql/schema`

### `execute_python`
Run Python code in the sandbox. Has access to `sql()`, `get_table()`, `pd`, `np`, `statsmodels`, `sklearn`, `plt`.

```json
{
  "code": "df = sql('SELECT hasDate, hasValue FROM LaborIndicator WHERE hasSeries = \"NJURN\"')\ndf['date'] = pd.to_datetime(df['hasDate'])\ndf.set_index('date')['hasValue'].plot()\nplt.title('NJ Unemployment Rate')",
  "timeout": 60
}
```
Returns: `{stdout, stderr, dataframes, figures, error}`.
Maps to: `POST /api/v1/execute`

### `get_series_data`
Fetch time-series data for a specific DataSeries ID. More efficient than SQL for direct series access.

```json
{"series_id": "NJURN"}
```
Returns: `[{date, value}]` sorted by date.
Maps to: `GET /api/v1/ontology/series/{id}/data`

### `generate_report`
Write a structured markdown report as an artifact. The report is stored via the storage service and returned as a download link.

```json
{
  "title": "NJ Labor Market Analysis Q4 2024",
  "content": "## Summary\n...\n## Methodology\n...\n## Findings\n...",
  "figures": ["<base64 PNG>"],
  "tables": [{"title": "Top Counties", "columns": [...], "rows": [...]}]
}
```
Returns: `{artifact_key, download_url}`.
Maps to: `POST /api/v1/analysis/run-code` with `upload_artifacts: true` (report rendered to PDF/HTML).

### `publish_to_github`
Commit changes (new configs, reports, analysis scripts) back to the project repo.

```json
{
  "files": [
    {"path": "analysis/unemployment_trends.py", "content": "..."},
    {"path": "reports/q4_2024_labor_market.md", "content": "..."}
  ],
  "commit_message": "Add Q4 2024 labor market analysis"
}
```
Returns: `{commit_sha, branch, url}`.
Maps to: `POST /api/v1/github/publish`

## Research Workflow Loop

The agent is designed to execute complete research workflows autonomously. A typical workflow:

```
User: "Analyze the relationship between housing prices and unemployment 
       across NJ counties over the last decade."

Turn 1 — Planning
  Agent reviews context snapshot:
    - LaborIndicator: 48,600 instances ✓
    - HousingIndicator: 12,300 instances ✓
    - County: 3,142 instances ✓
  Agent determines: data exists, can proceed to analysis

Turn 2 — Schema check
  [tool: get_sql_schema]
  Agent learns measuredFor_edges links indicators to counties
  Agent drafts SQL join strategy

Turn 3 — Data extraction
  [tool: run_sql]
  "SELECT c.hasName, l.hasValue as unemployment, h.hasValue as hpi,
          l.hasDate FROM County c
   JOIN measuredFor_edges mf1 ON mf1.object_iri = c._iri
   JOIN LaborIndicator l ON l._iri = mf1.subject_iri
   JOIN measuredFor_edges mf2 ON mf2.object_iri = c._iri
   JOIN HousingIndicator h ON h._iri = mf2.subject_iri
   WHERE l.hasDate = h.hasDate AND l.hasDate >= '2014-01-01'"
  → 12,400 rows returned

Turn 4 — Statistical analysis
  [tool: execute_python]
  OLS regression, correlation matrix, rolling correlation
  → DataFrame with results, 3 figures (scatter, time series, heatmap)

Turn 5 — Report generation
  [tool: generate_report]
  Full markdown report with embedded figures and summary table
  → artifact stored, download link returned

Turn 6 — Commit
  [tool: publish_to_github]
  Commits analysis script + report to project repo
  → commit SHA returned

Agent: "Analysis complete. Found significant negative correlation (r=-0.73)
        between HPI growth and unemployment in northern NJ counties 2019-2024.
        Full report: [link]. Committed to repo: [commit link]."
```

Total turns: 6. Total human interventions required: 0.

## SSE Event Protocol

The agent endpoint streams events via Server-Sent Events. The frontend consumes these to render the tool-call cards and streaming text in real time.

`POST /api/v1/agent/chat?project={slug}`

Request body:
```json
{
  "message": "Analyze housing vs unemployment in NJ counties",
  "history": [],
  "model": "claude-sonnet-4-6",
  "session_id": "session_abc123"
}
```

Response: `text/event-stream`. Each `data:` line is a JSON object:

| Event type | Fields | When |
|------------|--------|------|
| `text_delta` | `content: str` | LLM streaming text chunk |
| `tool_call` | `id, name, args: dict` | Agent calling a tool |
| `tool_result` | `id, name, result: any, error?: str` | Tool execution complete |
| `context_snapshot` | `snapshot: dict` | Context assembled for this turn (first event) |
| `done` | `new_messages: list` | Turn complete; full message list for history |
| `error` | `message: str` | Fatal error |

The `context_snapshot` event (sent once at the start of each conversation) lets the frontend display the "I have access to: N classes, M entities" card before the first text appears.

## Session Persistence

Agent conversations are persisted in the Convex `agentSessions` table, scoped to the project. The frontend passes `session_id` on subsequent messages to continue a conversation. History is stored as a flat list of `{role, content, tool_calls?, tool_call_id?}` message objects.

Sessions are browsable in the Agent page — the left panel shows previous conversations with auto-generated titles (derived from the first user message). Each session shows the project it belongs to.

## Multi-Turn Loop

The agent service runs a loop of up to `max_turns` turns (default 10, configurable per project). Each turn:

1. Assemble context snapshot (turn 1 only; subsequent turns reuse it).
2. Call LLM with current message history + tool schemas (filtered by `allowed_actions`).
3. Stream `text_delta` and `tool_call` events to client.
4. If tool calls present: execute each tool, stream `tool_result` events.
5. Append assistant message + tool results to history.
6. If no tool calls: stream `done` event and exit loop.
7. If `max_turns` reached: stream `done` with a note that the turn limit was hit.

Tool execution is sequential within a turn (tools may depend on each other's output). Tools across turns are independent.

---

## Action Validation

Before executing any tool call, the agent service validates:

1. **Allowed actions check** — tool name must be in the project's `allowed_actions` list. If not, the tool call is rejected with an error message returned as a `tool_result` event (the LLM sees the rejection and can adjust).

2. **Schema validation** — tool arguments are validated against the tool's JSON schema. Missing required fields or wrong types return a validation error as `tool_result`.

3. **Safety checks** per tool:
   - `run_sql`: query must be read-only (no `INSERT`, `UPDATE`, `DELETE`, `DROP`). Validated via simple SQL parse before execution.
   - `execute_python`: code is executed in the sandbox; `RAIL_EXECUTE_ENABLED` must be true.
   - `publish_to_github`: file paths must be within the project repo layout (no `../` traversal, no overwriting `rail.yaml`).
   - `create_data_source` / `create_pipeline`: content is validated via `yaml_service.validate()` before Convex mutation.

## Future: Platform-Wide Routing Agent

When multiple domain agents exist and cross-domain research is needed, a routing layer will:

1. Query the `Agent` class in the ontology to discover all available domain agents and their project contexts.
2. Parse the user's request to identify which domains are relevant.
3. Fan out sub-requests to the appropriate domain agents.
4. Merge and synthesize results.

This is built on top of domain agents, not instead of them. Domain agents remain the execution layer; the router handles only selection and synthesis. Design is deferred until at least two production domain agents exist.
